# gh で完結する Issue 駆動ワークフローを組んだ話

このリポジトリは、GitHub の Issue を起点にした開発フローを、できるだけ `gh`（GitHub CLI）の中で
完結させるために組んだ仕組みの実験場だ。ブラウザの Web UI を開かずに、ターミナルから
「Issue を起票する」「下書きを書く」「発行する」までを一本の流れにしている。

部品は3つある。

1. Markdown ベースの Issue テンプレート（`.github/ISSUE_TEMPLATE/`）
2. 人間と AI 双方が従う作業契約（`AGENTS.md` と `instructions/`）
3. 下書きの作成と発行を担う Claude Code のコマンド（`.claude/commands/`）

順番に何をしているか書いていく。

## なぜ作ったか

きっかけは Issue の質のばらつきだった。「ログインがおかしい」だけ書かれた Issue は、
着手前に必ず質問のラリーが発生する。再現手順は何か、どのファイルが対象か、何をもって
直ったと判断するのか。これを毎回やると、起票から着手までの時間がただ延びる。

そこで方針を一つ決めた。Issue は「追加質問なしで着手できる粒度」で書く。
対象ファイル、変更方針、受け入れ条件（テスト・検証方法）まで含める。
この方針を強制する入れ物として、テンプレートと契約ファイルを用意した。

## 1. Issue テンプレートは Markdown にした

最初は GitHub の Issue Forms（YAML）で作っていた。入力欄が構造化され、必須チェックも
効くので一見よさそうに見える。ただ実際に使うと制約がきつかった。フィールドの種類が
固定で、節を自由に増減できず、レイアウトもほぼいじれない。

結局 Markdown テンプレートに切り替えた。`.github/ISSUE_TEMPLATE/` には3種類置いてある。

- `feature.md` … 新機能・機能改善
- `bug.md` … 不具合の報告と修正依頼
- `docs.md` … ドキュメントの追加・修正

Markdown テンプレートの良いところは、本文が自由記述になることだ。見出しはそのまま使い、
要らない節は消し、HTML コメントで「ここに何を書くか」のヒントを埋め込める。
代わりに必須バリデーションとドロップダウンは失われるので、記述の質は書き手の責任になる。
今回はその責任を契約ファイル側で担保する設計にした。

各テンプレートの先頭には frontmatter があり、`title` の接頭辞（例 `feat(scope): `）と
`labels` を持たせてある。後述の発行コマンドはこの frontmatter を読んでタイトルとラベルを
決める。なお `config.yml` だけは Markdown ではなく設定ファイルで、空 Issue の禁止と
誘導リンクを定義している。

## 2. 作業のルールは AGENTS.md に集約した

`AGENTS.md` は、このリポジトリで人間と AI が作業するときの契約だ。迷ったらここを最優先する。
中身は重くない。要点だけ挙げると次のとおり。

- 作業前に `README.md` と `instructions/` の該当ファイルを読む
- 変更は1ステップずつ入れ、直後に検証する
- コミットは「1コミット＝1論点」、メッセージは `type(scope): subject`
- 作成・更新したファイルの先頭に概要コメントを書く（この README にも付けてある）

細かい規約は `instructions/` に分けた。コミットの書き方は `instructions/git-commit.md`、
GitHub の操作手順は `instructions/github-workflow.md`、設計ドキュメントの文体は
`instructions/doc-writing.md` にある。`github-workflow.md` には
「GitHub 操作には必ず `gh` を使う」「ブラウザ前提の手順を書かない」というルールがあり、
これが今回のワークフロー全体の土台になっている。この README 自体も `doc-writing.md` の
文体ルール（散文優先、設計判断は対案とトレードオフを併記、要約節を作らない）に従って書いている。

契約を1ファイルにまとめておくと、作業のたびに前提を説明し直さずに済む。
テンプレートが要求する「着手できる粒度」も、ここで裏打ちしている。

## 3. /ticket・/ticket-plan・/ticket-issue で下書きから発行まで

ここが今回の本体だ。Claude Code のカスタムコマンドを3つ用意した。下書きを作る系が
2つ（`/ticket` と `/ticket-plan`）、発行する系が1つ（`/ticket-issue`）という構成だ。

### /ticket — 会話から下書きを1件作る

`/ticket` を実行すると、まず種別（feature / bug / docs）を聞かれる。選ぶと対応する
テンプレートを土台に、`issues/feature-20260613-165320.md` のようにタイムスタンプ付きの
下書きが `issues/` に作られ、その絶対パスが表示される。

当初はテンプレートを丸ごとコピーするだけで、本文はパスをクリックしてエディタで人間が
書く設計だった。これをやめ、いまは**その時点までの会話内容から本文を生成する**ように
変えた。設計の相談をしてそのままチケットに落とす、という流れが実際の使い方の主流で、
空テンプレートを開いて手で書き写すのは二度手間だったからだ。

生成方式に変えると、AI が会話にない事実を埋めてしまう危険が出る。ここはコマンド側の
制約で抑えた。本文には会話で確定した事実だけを書き、不明な節は推測で埋めず `（未確定）`
の明示プレースホルダを置く。`title` も会話を要約した具体的なものに置き換え、`scope` の
ようなプレースホルダは残さない。機密情報・個人情報が会話に含まれていれば、下書きへ
転記する前に指摘してマスキングする。生成はあくまで下書きで、最終確認と編集は人間が
行う前提を崩していない。表示された絶対パスをクリックすれば、その場で確認・修正できる。

### /ticket-plan — 計画から下書きを複数件作る

`/ticket` は1コマンドで1下書きを作る。ところが設計の相談は、しばしば複数のチケットに
分かれる作業計画として固まる。これを `/ticket` に押し込もうとすると、出力が1件のときと
N件のときで保存先の命名も種別の聞き方も枝分かれし、コマンドの中が二重化する。

そこでコマンドを増やすか、`/ticket` に N 件モードを足すかで迷い、別コマンドに切り出す
方を採った。決め手は出力のカーディナリティ（1 か N か）が違うことだ。コマンドの境界は
入力ではなく出力の形で引くのが素直で、1件版と N件版を同居させると分岐が増えるだけだった。
代償として「下書きを作る」コマンドが2つに増え、利用者はどちらを使うか選ぶ必要が出る。
この負担は、種別選択の入口を分けることと、名前（`ticket` と `ticket-plan`）で吸収する判断にした。

`/ticket-plan` は、会話または引数で渡した計画ファイル（例 `@PLAN.md`）から作業項目を
抽出して一覧で示し、各項目に種別を自動で割り当てる。種別の確認は項目ごとに聞かず、
対応表をまとめて1回で取る。確定したら、タイムスタンプを1度だけ取得して全項目で共有し、
`issues/<type>-<timestamp>-NN.md`（`NN` は連番）の形で下書きを N 件生成する。最後に
全下書きの絶対パスを一覧表示する。

発行はこのコマンドに含めていない。N 件をまとめて発行すると、1件でも誤った下書きが
混ざったときに取り消しの効かない Issue が複数できてしまう。発行は1件ずつ `/ticket-issue`
に通す現状の手順を保ち、一括発行の便利さよりも誤起票の影響範囲を小さく保つ方を優先した。

### /ticket-issue — gh で発行する

下書きを書き終えたら `/ticket-issue @issues/feature-20260613-165320.md` のように
ファイルを渡す。コマンドは frontmatter から `title` と `labels` を取り出し、
frontmatter を除いた本文を Issue の body として、

```
gh issue create --title "<title>" --label "<labels>" --body-file <body>
```

を実行する。リポジトリは `-R` を付けず、カレントの origin を自動で使う。
`--title` `--label` `--body-file` をすべて指定するので、エディタも確認プロンプトも
挟まらず非対話で発行できる。発行前には、タイトルに `scope` のようなプレースホルダが
残っていないか、受け入れ条件が空でないかを確認する手順を入れてある。
Issue は外部に公開され取り消しが難しいので、ここだけは明示の確認を挟む。

`issues/` ディレクトリ自体は `.gitignore` で丸ごと除外している。下書きはスクラッチで、
リポジトリに残す対象ではないからだ。ディレクトリが無ければ `/ticket` が作り直す。

## 流れをまとめると

```
/ticket
  → 種別を選ぶ（feature / bug / docs）
  → 会話内容から本文を生成し issues/<type>-<timestamp>.md が作られる
  → 表示された絶対パスをクリックして内容を確認・修正

/ticket-plan [@PLAN.md]
  → 会話または計画ファイルから作業項目を抽出し一覧表示
  → 各項目に種別を割り当て（まとめて1回確認）
  → issues/<type>-<timestamp>-NN.md が N 件作られ、全パスを一覧表示

/ticket-issue @issues/<file>.md
  → frontmatter から title / label を取得
  → 本文を body にして gh issue create
  → 発行された Issue の URL が返る
```

起票の品質はテンプレートと契約で担保し、本文は会話から起こし、操作はすべて `gh` に
寄せた。ブラウザに戻る場面は基本的にない。

## hello world で一通り通した

コマンド群が机上で終わっていないことを確かめるため、Rust・C・Python の3言語で
hello world を作る計画を題材に、`/ticket-plan` から `/ticket-issue`、実装、検証までを
実際に流した。題材を hello world にしたのは、言語ごとのツールチェーンの差（cargo か、
コンパイル＋実行の二段か、スクリプト直実行か）が受け入れ条件にそのまま出る一方、
実装そのものでは詰まらず、コマンドの挙動だけを観察できるからだ。

`/ticket-plan` には3言語ぶんの開発計画を渡した。コマンドは作業項目を3件に分解し、
種別の確認を項目ごとに3回ではなく対応表1枚でまとめて取った（3件とも feature）。
タイムスタンプを1度だけ取得して共有する設計どおり、`issues/feature-20260613-175512-01.md`
から `-03.md` までが同一タイムスタンプ＋連番で衝突せずに生成された。発行はこのコマンドに
含めない方針も意図どおりで、生成は下書き止まりだった。

発行は `/ticket-issue` に1件ずつ通し、Issue #10（Rust）・#11（C）・#12（Python）が
立った。今回のリポジトリには `feature` ラベルが既にあったため、後述のラベル未登録には
当たっていない。frontmatter の `title` がそのままチケットのタイトルになり、`scope` の
プレースホルダは下書き生成の時点で具体名（`hello/rust` 等）へ置き換わっていた。

実装は共通設計どおり言語別サブディレクトリに分けた。`hello/rust/` は cargo プロジェクト、
`hello/c/` は C ソースと `make`／`make run`／`make clean` を持つ Makefile、
`hello/python/` は python3 で直接実行するスクリプトとした。検証環境は cargo 1.94.0、
Apple clang 21.0.0、Python 3.14.3。`cargo run`、`make run`、`python3 hello.py` の
いずれも標準出力に `Hello, world!` を出し、3件の受け入れ条件（ビルド成功・stdout 出力・
README への手順記載）を満たした。ビルド生成物（cargo の `target/`、C の実行バイナリ）は
リポジトリに残す対象ではないため `.gitignore` で除外している。

各言語の実行手順は次のとおり。

```
# Rust
cd hello/rust && cargo run

# C
cd hello/c && make run        # もしくは make してから ./hello

# Python
python3 hello/python/hello.py
```

## ハマったところ

きれいに動いたわけではない。実際に詰まった2点を残しておく。

**エディタのウィンドウを狙えない。** 最初は `/ticket` が下書きを Zed で自動的に開く
作りだった。ところが意図したウィンドウとは別のウィンドウで開いてしまう。
調べると、Zed の CLI は「呼び出したターミナルが属するウィンドウ」を指定できず、
最後にアクティブだったウィンドウに開く仕様だった。`--add` を付けても、狙った窓に
入るとは限らない。結局、自動起動はやめて絶対パスを表示するだけにした。
どのエディタでどう開くかは、クリックする人に任せたほうが確実だった。

**新しいリポジトリにはラベルが無い。** `/ticket-issue` を初めて流したとき、
`gh issue create --label feature` がラベル未登録で失敗した。テンプレートは
`feature` `bug` `documentation` のラベルが存在する前提だが、作りたての
リポジトリには何も無い。`gh label create` で3つ作って解決した。コマンドの
注意書きにも、発行前に `gh label list` で確認する手順を足してある。

## おわりに

派手な仕組みではない。テンプレートで Issue の粒度を揃え、契約で前提を固定し、
`gh` で操作を一本化しただけだ。ただ、起票から発行までブラウザを開かずに回せると、
日々の摩擦は確実に減る。同じように Issue の質と操作の手間で困っている人の参考になれば。
